Open Data - getting Data Using APIs

Open Data Access via APIs

Open Data

The Foundations for Evidence-Based Policymaking Act of 2018, also known as the Open Data Law, became law on January 14, 2019. It requires U.S. federal agencies to publish their information online as standardized, machine‑readable open data, with metadata listed in the Data.gov catalog.

  • Summary

The Open Data Law makes government information easier for everyone to find, use, and analyze. It ensures that data is shared in clear formats so students, researchers, and the public can work with it.

Why We Learn Open Data After Tidyverse and GitHub

After learning Tidyverse (to clean and analyze data) and GitHub (to store and share your work), the next step is learning where real data comes from.
Open Data gives us free, public information from governments and organizations.
APIs let us pull that data automatically into R.

Together, they help us: - work with real‑world, up‑to‑date information
- build reproducible workflows
- connect our Tidyverse skills to live data sources
- create projects that update themselves

Goal of This Part

The goal is to learn how to find, access, and import real data into R using APIs, so we can apply our Tidyverse skills to meaningful, real‑world datasets.

Where Open Data Comes From

  • Federal Government: Over 350,000 datasets listed on Data.gov, covering many topics (economy, health, environment, transportation, etc.).

  • Federal Agency Sites: Many agencies have their own open data portals (e.g., Census Bureau, USDA, Department of Education, HHS).

    • Federal Agencies and Their Open Data Sites (in simple words)

      • Most federal departments and agencies have their own websites where they share open data, and many also list their data in the main Data.gov catalog. These sites usually include tools to explore or visualize the data and offer downloads in different formats such as CSV, Excel, HTML, ZIP files, or special mapping formats.

      • Some agencies let you download data directly through your browser, while very large datasets may require FTP. A few agencies have highly specialized data portals, while others rely mainly on Data.gov to share their information.

  • Local Governments: Cities like Baltimore, New Orleans, NYC, San Francisco, and Washington DC publish open data.

  • International Sources: United Nations, World Bank, OECD, Open Data Portal Europe, and others.

  • Organizations: Some groups curate open data (e.g., Johns Hopkins COVID‑19, Kaggle, Awesome Public Data Sets).

Bottom Line

There is open data everywhere — billions of bytes available for public use — and learning how to access it is essential for modern data work.

Types of Web Data

The Web as an Iceberg (in simple words)

If you think of the web as an iceberg, Open Data is the small part above the water—the surface web. This is where search engines and web crawlers can freely find information.

But most data is below the surface, in the deep web. This data is protected behind firewalls, logins, or APIs, so search engines (and you) cannot access it without extra steps to prove who you are.

Organizations protect their data because: - they want to know who is using it
- they want to prevent overuse
- they need to protect security, privacy, and intellectual property

Below the deep web is a small portion called the dark web, which requires special tools to access.
We will not work with the dark web at all.

How We Access Data on the Web

There are four main ways to get data from the web:

  1. Download a file (CSV, Excel, JSON, XML, etc.) by clicking a link.
  2. Use an R package that connects to an API.
  3. Write your own code to interact directly with an API.
  4. Scrape data from an HTML webpage.

We already know how to do #1.

API

  • API stands for Application Programming Interface. APIs are like bridges that allow different software systems to communicate and share data or features with each other

  • Organization use a software to act as a gateway to control the access to data. So, they use APIs.

What is a common form of API?

  • A common form of API Known as REpresentational Slate Transfer or REST API.

  • It is used to enable communication between different software applications by providing a standardized way to access and exchange data over the internet,

What does API do?

  • It can control
    1. Who can access
    2. What can be accesed
    3. How can it be accesed
    4. When it can be accessed
    5. What is returned

What is API ENDPOINT?

  • It is a location of resources to be accessed by an API.(Think of it as a URL)

  • API may act as GATEWAYS to many different ENDPOINTS which can be turned open or closed.

What is API DESCRIPTION document?

  • Organization typically published an API description document to provide users information on how to interact with API.

  • Question: How to find the API documentation for a website?

    • Easiest way is to use google: Use a query like: “API documentation site:example.com” (replacing example.com with the actual website domain).

API’s Formats

What are the most two formats that API will return the data?

These days, most API will return data in form of

  1. XML or eXtensible Markup Language
  • For storing, transforming, and reconstructing arbitrary data.
  • Arbitrary data are such as, text, image, audio, video, etc.
  1. JSON or Java Script Object Notation
  • It is relatively easier/simpler than other format (or a lightweight data interchanging format)
  • It is easier for human to read and write and easier for machine to parse(process) and generate.

An API is like a user interface for computers.

  • When you click a button on a website, you expect something to happen.

  • When a computer program uses an API, it’s like the program is “clicking a button” and asking another system for information.

  • APIs hide the complicated code behind the scenes. They make it easy for different programs to talk to each other, no matter what programming language they use.

  • There are many free and public APIs available here: https://github.com/public-apis/public-apis

What is an API wrapper?

  • An API wrapper is a helper package that makes an API easier to use.

  • Instead of writing long, complicated code to talk to an API, a wrapper gives you simple functions that do the hard work for you.

Think of it like this:

  • The API is the machine.

  • The wrapper is the set of buttons that make the machine easy to use.

  • You press a button → the wrapper talks to the API → you get clean data back.

  • You don’t need to worry about URLs, authentication, or formatting — the wrapper handles all of that behind the scenes.

Authentication

Not all APIs are open. Many want to know who is using their data.
Different APIs use different levels of security.

1. No Authentication

Some APIs are completely open (like the arXiv API).
You can use them without logging in or identifying yourself.

2. Basic Authentication

Some APIs ask for a username and password.

Example:

GET("url", authenticate("username", "password"))
Error in GET("url", authenticate("username", "password")): could not find function "GET"

You usually apply for an account, then keep your password safe.

API Keys and Personal Access Tokens (PATs)

  • Many modern APIs require an API key or Personal Access Token.

  • This is like a long, unique password that identifies you.

  • You may need to register as a developer.

  • Your key is tied to your identity.

  • You are responsible for anything done with your key.

  • Never put your key in a file you might share (like .R scripts, .qmd, or .Rmd files).

How to Store Your Key Safely

There are two common ways to store your key so R can use it:

  1. .Renviron file

  • Easy to set up

  • But the key is stored as plain text (not protected)

  1. keyring package

  • Stores your key securely

  • Uses your computer’s password manager

  • Safer than .Renviron

Why APIs Use Authentication

Organizations use different levels of control depending on how sensitive their data is:

  • Some only ask for an email.

  • Some require a username and password.

  • Many require API keys or PATs.

  • Some require full developer registration and give you multiple keys for different apps.

Each API explains its rules in its documentation — and you must read it, because every API works a little differently.

Once you get an API key or PAT, you are responsible for everything done with it.

Never save or display your key in a file you might share.

Always read the API documentation because each API works differently.

Getting an API Key

APIs often require a special code called an API key.
This key identifies you and gives you permission to use the API.

Below is an example of how to get and store an API key for the Open Movie Database (OMDb).

Getting an API Key for the Open Movie Database (OMDb)

OMDb is the Open Movie Database.
It provides information about movies and TV shows.
It is free to use, but you must request an API key.

How to Get an OMDb API Key

  1. Go to the OMDb website: https://www.omdbapi.com/
  2. Click API Key at the top of the page.
  3. Choose the FREE option.
  4. Enter your email, name, and how you plan to use the API.
  5. OMDb will email your key (usually within an hour).
  6. Keep the email — you cannot recover the key if you lose it.

Make sure to look for the link to verify your email and/or activate the key.

Option 1 (Easy but Not Very Secure): Store Your Key in .Renviron

  • Using the .Renviron file is an easy way to store your API key or Personal Access Token (PAT).

  • It is safer than putting your key directly into a .qmd or .R file that you might share,
    but it is not fully secure because the key is still stored as plain text.

Steps to Store Your OMDb Key in .Renviron

  1. Make sure you have the usethis package installed
    (it comes with the devtools package).

  2. Open your .Renviron file from the R console:

usethis::edit_r_environ()
  1. When the .Renviron file opens, add a line like this:
OMDB_API_KEY = your-private-key-here
Error: object 'your' not found

OMDB_API_KEY is the name you choose for your key.

  • your-private-key-here is the key OMDb emailed to you.
  1. Save the file and restart R.

Accessing Your Key in R

Once your key is stored in .Renviron, you can access it anytime with:

Sys.getenv("OMDB_API_KEY")
[1] "8fa7cd7e"

This lets you use the key in your code without typing it directly into your script, which helps keep it private.

Option 2 (More Secure): Using the {keyring} Package

Storing your API key in .Renviron is easy, but it is still a plain text file, which means the key is not fully protected.
A more secure option is to use the keyring package.

The keyring package stores your key inside your computer’s secure credential manager, the same place your computer stores passwords.

Learn more here: https://github.com/r-lib/keyring

Why use keyring?

  • Your key is stored securely, not in plain text.
  • Your key is protected by your computer’s login password.
  • Your key is not visible in .Rhistory, .qmd, or .R files.
  • You can access the key safely inside your R code.

Step-by-Step: Store Your OMDb Key with keyring

  1. Install and load the package (if you don’t have it):

install.packages("keyring") and library(keyring)

  1. Save your key securely (do this once):
key_set("OMDB_API_KEY_SECURE")
Error in key_set("OMDB_API_KEY_SECURE"): could not find function "key_set"

  • A pop‑up window will appear.

  • Paste the API key you received from OMDb.

  • If asked, choose Allow Always.

  1. Retrieve your key in your R code:
key_get("OMDB_API_KEY_SECURE")
Error in key_get("OMDB_API_KEY_SECURE"): could not find function "key_get"

This returns your key without exposing it in your script. Your key is not exposed either in the file or your .Rhistory

.Rhistory is a plain text file where R automatically saves the commands you typed in the Console.
That’s it — only your commands, not the results.
  • **Note: For Mac Users. You may have to go to your Mac’s KeyChain access utility to change the permission under login passwords to “allow all aplications to access this item” so it works in RStudio,

For Mac Users

🔑 Managing API Keys on macOS

Starting in 2023, Apple introduced the Passwords & Passkeys app as the main place for users to view saved passwords.
However, this app does not allow programs (like R) to access stored API keys.

macOS still uses Keychain Access as the secure system for storing secrets such as API keys.
Keychain Access supports programmatic access, which means tools like R, Python, or Swift can retrieve keys safely.

🗝️ Viewing and Editing API Keys in Keychain Access

  • Open Keychain Access
    Press ⌘ + Space, type Keychain Access, press Enter.
  • (Optional) Keep it handy
    Right‑click the dock icon → Keep in Dock.
  • Find your API key
    Look under login → Passwords.
    Double‑click the item → check Show password (enter your Mac password).

⚙️ Adjusting Permissions for Apps (e.g., RStudio)

In the key’s detail window:

  • Go to Access Control
  • Choose one of the following:
    • Allow all applications to access this item (less secure), or
    • Add RStudio.app (recommended)

This allows RStudio to retrieve the key automatically when your script calls key_get().

Warning

Anyone who has your computer password can open your credential manager and see your key.

Even without your password, someone who has access to your computer and knows R and the {keyring} package could still retrieve your key.

However, using the credential manager with {keyring} is much more secure than storing your key in a plain text file like .Renviron.

Using Your Key in Your Code

When you need to use your API key in your code, you have two choices:

Option 1: Save the key to a variable (NOT recommended)

You could write something like:

my_key <- key_get("OMDB_API_KEY_SECURE")
Error in key_get("OMDB_API_KEY_SECURE"): could not find function "key_get"

But this is not a good idea because:

  • The key will appear in your global environment

  • The key may be saved in your .Rhistory file (Rhistory stores everything you typed in the Console)

This means your key could accidentally be exposed.

Option 2: Use the key directly inside your API call (recommended)

A safer approach is to never store the key in a variable. Instead, call it directly inside the function that needs it.

Examples:

Using keyring:

x <- GET(
  url = "URL",
  apikey = key_get("OMDB_API_KEY_SECURE")
)
Error in GET(url = "URL", apikey = key_get("OMDB_API_KEY_SECURE")): could not find function "GET"

Using .Renviron:

x <- GET(
  url = "URL",
  apikey = Sys.getenv("OMDB_API_KEY")
)
Error in GET(url = "URL", apikey = Sys.getenv("OMDB_API_KEY")): could not find function "GET"

This way:

  • Your key is never saved in the environment

  • Your key is never written to .Rhistory

  • Your key stays protected

mout <- GET(
  url = "url",
  apikey = key_get("my_credential")
)

Using the {httr2} Package to Access APIs

Most APIs on the internet use HTTP (Hyper‑Text Transfer Protocol).
HTTP is simply the language that computers use to request and send data.

You do not need to learn HTTP itself for this course, but you will use R’s {httr2} package to communicate with APIs through HTTP.

Common HTTP Verbs

These are the six most common HTTP actions:

  • GET() – request data
  • POST() – send data
  • PUT() – update data
  • PATCH() – partially update data
  • DELETE() – remove data
  • HEAD() – check if something exists

We will mostly use GET, because most public APIs are read‑only.

Each API has its own rules.
You must read the documentation to understand:

  • What parameters you can send
  • How to send them
  • What data you should expect back

There is no universal standard — every API works a little differently.

The {httr2} Package

{httr2} is the modern replacement for the older {httr} package, which is no longer updated.

What makes {httr2} different?

  • It uses a request object instead of running GET() immediately.
  • You can build your request step‑by‑step.
  • You can inspect or modify the request before sending it.
  • It uses {curl} behind the scenes for low‑level HTTP work.

This makes API work easier to understand and debug.

Install and Load {httr2}

Install.packages("httr2") and library(httr2)

Building a Request Object

When using {httr2} to call an API, you build a request object step by step.
This lets you see and modify your request before actually sending it.

1. Start with the URL

Every API request begins with a base URL.

For OMDb, the documentation says all requests go to:

http://www.omdbapi.com/?apikey=[yourkey]&

Everything after the ? is a query parameter, so we start with just the base URL:

my_url <- "http://www.omdbapi.com/"
req <- httr2::request(my_url)
req
<httr2_request>
GET http://www.omdbapi.com/
Body: empty

The Body: empty output indictaed that the request exists, but we haven’t added any details yet.

  • Dry‑run the request

A dry run shows what would be sent, without actually sending it:

req |> httr2::req_dry_run()
GET / HTTP/1.1
accept: */*
accept-encoding: deflate, gzip
host: www.omdbapi.com
user-agent: httr2/1.1.2 r-curl/6.3.0 libcurl/8.14.1

This prints the HTTP method (GET), headers, and host.

2. Add Query Parameters

APIs use name = value pairs to tell the server what you want.

OMDb allows parameters like:

  • t \(\to\) movie title

  • plot \(\to\) short or full

  • r \(\to\) return format

  • apikey \(\to\) your API key

We add them one at a time:

              t = "Inception"
              )

# Make the GET request to the OMDb API
response <- GET(url = URL, 
                query = params
                )

# Check if the request was successful

  # Parse the JSON response into an R list
  movie_data <- content(response, as = "parsed")

  print(paste("Title:", movie_data$Title))
  print(paste("Year:", movie_data$Year))
  print(paste("Director:", movie_data$Director))
  print(paste("Plot:", movie_data$Plot))
} else {
  # Print an error message if the request failed
  print("Error: Could not retrieve data.")
}

R Packages to “Wrap” APIs

When you work with data from the web, many websites and databases provide an API—a structured way for your code to request data.
But writing raw API calls can feel complicated, especially when you’re just starting out.

That’s why many developers create R packages that “wrap” APIs.
These packages give you easy-to-use R functions that handle the API work behind the scenes.

Why These Packages Are Helpful

  • They use familiar R syntax, so you don’t have to manually build URLs or handle authentication details.
  • They return data in R-friendly formats like data frames or tibbles.
  • They save time and reduce errors.
  • They often include helpful documentation and examples.

Tip

If an R package already exists for the API you want to use, start with that package.
It will almost always be easier and faster than writing your own API calls.

If the package doesn’t include everything you need, you can look at its source code to learn how it interacts with the API. Then you can build your own custom functions using the API’s documentation.

Examples of R Packages That Wrap APIs

Here are some popular R packages that make API data easy to access:

What the {tidycensus} Package Does

{tidycensus} is an R package that makes it easy to download data from the U.S. Census Bureau.
Instead of writing complicated API code, you can use simple R functions to get clean data frames.

In simple words:

  • The Census Bureau has a lot of data stored behind special web addresses called API endpoints.
  • {tidycensus} knows where these endpoints are and how to talk to them.
  • You ask for data using an R function, and the package brings it back already cleaned and ready to use.

It can also return map-ready data, which is helpful for making geographic visualizations.

What Is an API Endpoint?

An endpoint is just a specific URL where a dataset lives.

Example:

One endpoint might give population numbers.
Another might give housing information.

{tidycensus} knows how to connect to many of these endpoints for you.

Types of Census Data You Can Access

1. Decennial Census (every 10 years)

  • It is a complete population count that happens every 10 years in the U.S. It gathers basic information, like age, race, and housing, from every household to help plan for things like schools, roads, and public services.

Examples of available files:

  • 2000 Census Summary Files
  • 2010 Census Summary Files
  • 2020 Redistricting Data

Note: As of July 2023, only some 2020 Census data has been released.

2. ACS (American Community Survey)

  • It is an ongoing survey that collects more detailed data on topics like education, income, and employment. Instead of every 10 years, it samples a smaller group of people every year to give communities updated information between censuses.

The ACS is a survey, not a full count.
It collects data every year from about 3 million households.

Because it’s a sample:

  • The numbers are estimates
  • Each estimate has a margin of error (MOE)

get_acs() always returns both.

Other R Packages for Census Data

  • {censusapi} — gives access to even more Census APIs
  • {ipumsr} — works with NHGIS data
  • Many others listed in “A Guide to Working with US Census Data in R”

Using JSON Structured DATA

  • Java Script Object Notation
  • JSON uses conventions familia to programmers for many languages so it is an ideal data interchange language. for example
{
  "name": "John",
  "age": 30
}

  • JSON uses a simple format with keys and values in curly braces {}.

  • In programming:

    • Key: The name or label (e.g., “name”).
    • Value: The information or data assigned to that key (e.g., “John”).
    • A key and value separated by a colon :
      • Example: "age": 30

JSON-R Package

There are multiple packages working with JSON

  • {jsonlite} package is a JSON parser/generator optimized for the web. It has bidirectional mapping between JSON data and the most important R data types.

    • Simplification is process by which JSON arrays are automatically converted from a list into more specific R class.

    • The fromJSON() function has 3 arguments which control the simplification process:

      • simplifyVector
      • simplfyDtatFrame
      • simplifyMatrix
  • The {tidyjson} package takes an alternate approach to structuring JSON data into tidy data
